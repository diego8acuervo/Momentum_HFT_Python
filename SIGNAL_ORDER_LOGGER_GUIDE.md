# Signal and Order Logger - Complete Guide

## 📊 Overview

The **SignalOrderLogger** helps you debug quantity calculation issues by tracking the entire signal → order pipeline. It captures:

1. **Signals** generated by the strategy (with hedge ratios and z-scores)
2. **Orders** created by the portfolio (with all calculation details)
3. **Quantity calculations** step-by-step

This lets you identify where oversized orders are coming from.

## 🎯 Purpose

You reported that orders are too large for the available capital. This logger will help you find:
- Is the **signal quantity** wrong (strategy calculation)?
- Is the **hedge ratio** calculation incorrect?
- Is **calcular_unidades()** producing wrong values?
- Is the **ATR calculation** using bad historical data?
- Is **risk management** being applied correctly?

### Automatic Operation

```bash

```

### What Gets Logged

#### 1. Signal Events (From Strategy)
When strategy generates a signal, captures:
- `timestamp` - When signal was generated
- `symbol` - Trading symbol (ETH, XRP, etc.)
- `signal_type` - Type of signal (LARGO, CORTO, FUERA)
- `signal_quantity` - Quantity from strategy (THIS IS KEY!)
- `z_score` - Z-score that triggered the signal
- `hedge_ratio` - Hedge ratio for the pair

#### 2. Order Events (From Portfolio)
When portfolio creates an order, captures:
- `timestamp` - When order was created
- `symbol` - Trading symbol
- `direction` - Order direction (buy/sell)
- `order_type` - MKT, LMT, etc.
- `order_quantity` - Final quantity in the order (THIS IS KEY!)
- `order_price` - VWAP or close price used
- `signal_type` - Original signal that triggered this
- `cash_available` - Cash available at time of order
- `price_used` - Price used in calculation
- `notional_value` - order_quantity × price
- `qty_to_cash_ratio` - Ratio of notional to cash (>1.0 = oversized!)

## 📊 Output Files

When you stop the program (Ctrl+C), two CSV files are created:

### 1. Signals Log
**Filename**: `signals_SYMBOL1_SYMBOL2_YYYYMMDD_HHMMSS.csv`

**Columns**:
- `timestamp` - When signal generated
- `symbol` - Which asset
- `signal_type` - LARGO/CORTO/FUERA
- `signal_quantity` - Quantity from strategy  
- `z_score` - Z-score value
- `spread` - Spread value (if available)
- `hedge_ratio` - Hedge ratio used

### 2. Orders Log
**Filename**: `orders_SYMBOL1_SYMBOL2_YYYYMMDD_HHMMSS.csv`

**Columns**:
- `timestamp` - When order created
- `symbol` - Which asset
- `direction` - buy/sell
- `order_type` - MKT, etc.
- `order_quantity` - **FINAL QUANTITY**
- `order_price` - Price used
- `signal_type` - Original signal type
- `cash_available` - Cash at order time
- `price_used` - Price in calculation
- `atr_value` - ATR if used (optional)
- `risk_pct` - Risk % if used (optional)
- `raw_quantity` - Before rounding (optional)
- `final_quantity` - After rounding
- `notional_value` - Total order value
- `qty_to_cash_ratio` - **KEY METRIC!**

## 🔧 Finding the Root Cause

### Check 1: Signal Quantities

Look at signals CSV - are the quantities reasonable?

```python
signals = pd.read_csv('outputs/signals_ETH_XRP_20251203_120000.csv')
print(signals[['symbol', 'signal_type', 'signal_quantity', 'hedge_ratio']])
```

**Expected for pairs trading**:
- Base asset (ETH): `signal_quantity = 1.0`
- Hedge asset (XRP): `signal_quantity = hedge_ratio` (e.g., 63.77)

**If wrong**: Problem is in strategy's `calcular_senalXY()` method

### Check 2: Hedge Ratio

Is the hedge ratio reasonable?

```python
print(f"Average hedge ratio: {signals['hedge_ratio'].mean()}")
print(f"Max hedge ratio: {signals['hedge_ratio'].max()}")
print(f"Min hedge ratio: {signals['hedge_ratio'].min()}")
```

**Expected**: Usually between 0.1 and 100 depending on price ratios
**If wrong**: Problem is in OLS regression calculation

### Check 3: Order Quantities vs Signal Quantities

Compare signals to orders:

```python
# Merge signals and orders
merged = pd.merge(
    signals[['timestamp', 'symbol', 'signal_quantity']],
    orders[['timestamp', 'symbol', 'order_quantity']],
    on=['timestamp', 'symbol'],
    how='inner'
)

merged['quantity_multiplier'] = merged['order_quantity'] / merged['signal_quantity']
print(merged[['symbol', 'signal_quantity', 'order_quantity', 'quantity_multiplier']])
```

**What to look for**:
- If `quantity_multiplier` is huge (>1000x), then `calcular_unidades()` is amplifying too much
- This suggests ATR-based sizing is using wrong values

### Check 4: Cash Management

```python
print(df.groupby('symbol').agg({
    'cash_available': 'mean',
    'notional_value': 'mean',
    'qty_to_cash_ratio': 'mean'
}))
```

**Expected**: `qty_to_cash_ratio` should be < 0.5 for conservative trading
**If >> 1.0**: Orders are way too large!

## 🎓 Common Issues and Solutions

### Issue 1: Signal Quantity = Hedge Ratio (63.77)

**Problem**: Both legs using hedge ratio as quantity
**Solution**: Base leg should be 1.0, only hedge leg should be hedge_ratio

```python
# In Estrategia.py, check:
senal_y = EventoSenal(1, p0, dt, 'LARGO', 1.0)      # ← Should be 1.0
senal_x = EventoSenal(1, p1, dt, 'CORTO', hr)       # ← Should be hr
```

### Issue 2: calcular_unidades() Returns Huge Values

**Problem**: ATR-based sizing using wrong historical data
**Check**: 
```python
# In PortAQMHFT.py, add debug logging
cantidad_mdo = self.calcular_unidades(senal)
print(f"DEBUG: calcular_unidades returned {cantidad_mdo} for {senal.nemo}")
```

**Solution**: Verify ATR calculation is using correct period and data

### Issue 3: Orders Exceed Cash

**Problem**: No cash constraint in quantity calculation
**Solution**: Add cash check in `genera_orden()`:

```python
cantidad_mdo = self.calcular_unidades(senal) * fuerza

# Add cash constraint
max_affordable = self.cuenta_actual['Caja'] / (2 * p_mdo)  # Divide by 2 for both legs
cantidad_mdo = min(cantidad_mdo, max_affordable)
```

### Issue 4: Hedge Ratio Too Large

**Problem**: OLS regression with mismatched price scales
**Example**: ETH ($3000) vs XRP ($2) gives huge hedge ratios

**Solution**: 
1. Check if pairs are cointegrated properly
2. Consider using normalized prices
3. Use log prices instead of raw prices

## 📈 Analysis Example

Here's a complete analysis workflow:

```python
import pandas as pd
import matplotlib.pyplot as plt

# Load data
signals = pd.read_csv('outputs/signals_ETH_XRP_20251203_120000.csv')
orders = pd.read_csv('outputs/orders_ETH_XRP_20251203_120000.csv')

# 1. Check hedge ratios
print("=== HEDGE RATIO ANALYSIS ===")
print(f"Mean: {signals['hedge_ratio'].mean():.2f}")
print(f"Std: {signals['hedge_ratio'].std():.2f}")
print(f"Range: {signals['hedge_ratio'].min():.2f} - {signals['hedge_ratio'].max():.2f}")

# 2. Check order sizes
print("\n=== ORDER SIZE ANALYSIS ===")
print(orders.groupby('symbol')['order_quantity'].describe())

# 3. Check cash ratios
print("\n=== CASH RATIO ANALYSIS ===")
print(f"Orders within cash: {len(orders[orders['qty_to_cash_ratio'] <= 1.0])}/{len(orders)}")
print(f"Max overage: {orders['qty_to_cash_ratio'].max():.2f}x")

# 4. Plot notional vs cash
fig, (ax1, ax2) = plt.subplots(1, 2, figsize=(14, 5))

# Plot 1: Notional values
orders.groupby('symbol')['notional_value'].mean().plot(kind='bar', ax=ax1)
ax1.axhline(y=1000, color='r', linestyle='--', label='Available Cash')
ax1.set_title('Average Notional Value per Symbol')
ax1.set_ylabel('Notional Value ($)')
ax1.legend()

# Plot 2: Quantity ratios
orders.groupby('symbol')['qty_to_cash_ratio'].mean().plot(kind='bar', ax=ax2)
ax2.axhline(y=1.0, color='r', linestyle='--', label='Cash Limit')
ax2.set_title('Average Quantity-to-Cash Ratio')
ax2.set_ylabel('Ratio')
ax2.legend()

plt.tight_layout()
plt.savefig('outputs/order_size_analysis.png')
print("\n📊 Plots saved to outputs/order_size_analysis.png")
```
